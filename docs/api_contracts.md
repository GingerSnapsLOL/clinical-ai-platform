## API Contracts

This document summarizes the main API contracts defined in `services/shared/schemas_v1.py`.  
The source of truth is always the Pydantic models in that module.

---

## Conventions

- All request/response models include a `trace_id` field for end-to-end traceability.
- All services use a shared `Status` literal:
  - `"ok"` – successful operation.
  - `"error"` – error envelope with `ErrorInfo`.
- Errors are represented as:
  - `ErrorInfo` with:
    - `code: str`
    - `message: str`
    - `details: dict | None`

## Health

### Response

- **Model**: `HealthResponse`
- **Fields**:
  - `status: "ok"`
  - `service: str` – service name.
  - `version: str` – currently `"0.1.0"`.

All services expose `GET /health` returning `HealthResponse`.

## Gateway / Orchestrator – `/v1/ask`

### Request (internal)

- **Model**: `AskRequest`
- **Fields**:
  - `trace_id: str` – UUID generated by gateway when omitted by the client.
  - `mode: Mode` – `"strict"` or `"hybrid"` (currently only strict behavior is implemented).
  - `note_text: str` – clinical note text.
  - `question: str` – user question.
  - `user_context: dict | None` – optional extra context.

### Response

- **Model**: `AskResponse`
- **Fields**:
  - `status: Status` – `"ok"` or `"error"`.
  - `trace_id: str`
  - `pii_redacted: bool` – whether PII redaction was applied.
  - `answer: str` – synthesized answer (LLM or deterministic fallback).
  - `sources: list[SourceItem]` – evidence passages.
  - `entities: list[EntityItem]` – extracted entities from NER.
  - `risk: RiskBlock | None` – risk score + explanation (currently stubbed).
  - `warnings: list[str]` – non-fatal pipeline warnings.
  - `error: ErrorInfo | None` – populated when `status == "error"`.

### Supporting types

- `SourceItem`
  - `source_id: str`
  - `title: str | None`
  - `url: HttpUrl | None`
  - `snippet: str | None`
  - `score: float`
  - `metadata: dict | None`

- `CitationItem`
  - `source_id: str`
  - `title: str | None`
  - `url: HttpUrl | None`

- `EntityItem`
  - `type: str`
  - `text: str`
  - `start: int`
  - `end: int`
  - `confidence: float | None`

- `FeatureContribution`
  - `feature: str`
  - `contribution: float`

- `RiskBlock`
  - `score: float`
  - `label: "low" | "medium" | "high"`
  - `explanation: list[FeatureContribution]`

## LLM Service – `/v1/generate`

The LLM service defines request/response models in `services/llm-service/app/main.py` (not via `schemas_v1.py`).  
The orchestrator calls this API via `services.shared.llm_client.LLMClient` (env `LLM_BASE_URL`, e.g. `http://llm-service:8060` in Docker).

### `POST /v1/generate`

**Request JSON** (`GenerateRequest`):

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `trace_id` | `string` | yes | Same UUID as the rest of the pipeline; echoed in the response and used for logs. |
| `prompt` | `string` | yes | Single text bundle from the orchestrator (`_build_llm_prompt` in `services/orchestrator/app/main.py`: question, entities, top evidence passages, risk block, instructions). |
| `max_tokens` | `int` | no | Maximum **new** tokens to generate (default `512`). |
| `temperature` | `float` | no | Sampling temperature (default `0.2`). |

**Response JSON** (`GenerateResponse`):

| Field | Type | Description |
|-------|------|-------------|
| `status` | `Status` | `"ok"` on success. |
| `trace_id` | `string` | **Must match** request `trace_id` (server does not mint a new id). |
| `text` | `string` | Decoded completion (new tokens only; not the full prompt). |
| `usage` | `object` | `prompt_tokens`, `completion_tokens`, `total_tokens` (counts derived from tensors in the handler). |

### `trace_id` propagation (orchestrator → llm-service)

1. **JSON body**: `LLMClient.generate()` sends `trace_id` in the POST body as `LLMGenerateRequest`.
2. **Header**: the same client sets **`X-Trace-Id`** (`TRACE_ID_HEADER` in `services/shared/http_client.py`).
3. **Handler**: `/v1/generate` calls `set_trace_id(request.trace_id)` so structured middleware logs include the trace.
4. **Response**: `GenerateResponse.trace_id` is copied from `request.trace_id`.

### Prompt shaping and “In 100 words.” (or similar) at the start of output

- **Not in repository text**: neither `services/orchestrator/app/main.py` (`_build_llm_prompt`) nor `services/llm-service/app/main.py` contains the literal phrase “100 words”. The orchestrator asks for a **concise, clinician-facing** answer; `llm-service` does not prepend fixed words to the completion.
- **Source of the artifact**: wording like “In 100 words.” is produced by the **model-derived continuation** (`model.generate` → `tokenizer.decode` on **new** tokens only). It often appears when an **instruct/chat** model sees a long, instruction-heavy block as **raw text** instead of the tokenizer’s **chat template** (Special tokens / roles trained into Qwen).
- **Mitigation in `llm-service`**: when the tokenizer provides `apply_chat_template`, the orchestrator prompt is wrapped as a **user** message and formatted with `add_generation_prompt=True` before encoding (see `app/main.py`). That aligns the input with how Qwen2.5-Instruct was trained and reduces meta-preambles.

## Error handling patterns

- Services should:
  - Use `status="ok"` and omit `error` on success.
  - Use `status="error"` and populate `error: ErrorInfo` when returning a logical error envelope.
- The gateway wraps orchestrator/network errors in a JSON error body containing:
  - `status: "error"`
  - `trace_id: str`
  - `error: ErrorInfo`

For the authoritative details and any future changes, refer to `services/shared/schemas_v1.py` and individual service `app/main.py` files.
